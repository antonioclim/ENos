Why Text Processing?

In Linux, everything is a file and most files are text:

  • Configuration files (/etc/*)
  • System and application logs
  • Source code
  • CSV, JSON, XML data
  • Command output
Superpower: With grep, sed and awk you can process in seconds what would take others hours!

Regular Expressions - Foundation

Regex = a language for describing patterns in text

# Simple pattern - search for the text "error"
error

# Pattern with metacharacters - any line starting with #
^#.*
‚ö†Ô∏è Regex ‚â† Globbing!
‚Ä¢ Globbing (shell): *.txt = files with the .txt extension
‚Ä¢ Regex: .*\.txt = text ending with .txt

Basic Metacharacters

Symbol Meaning Example
. Any character (exactly 1) a.c ‚Üí abc, a1c, aXc
^ Start of line ^Hello ‚Üí lines starting with Hello
$ End of line end$ ‚Üí lines ending with end
* 0 or more (preceding) ab*c ‚Üí ac, abc, abbc
\ Escape special character \. ‚Üí literal dot

BRE vs ERE - The CRITICAL Difference

BRE (grep default)
  • \+ = 1 or more
  • \? = 0 or 1
  • \{n,m\} = interval
  • \( \) = grouping
ERE (grep -E)
  • + = 1 or more
  • ? = 0 or 1
  • {n,m} = interval
  • ( ) = grouping
üö® Common mistake: grep 'ab+c' searches literally for "ab+c"!
Correct: grep -E 'ab+c' or grep 'ab\+c'

GREP - Global Regular Expression Print

Searches for patterns in files and displays matching lines.

$ grep 'pattern' file.txt
the line containing pattern
Variants:
  • grep - BRE (basic)
  • grep -E / egrep - ERE
  • grep -F / fgrep - fixed string
Format:
  • grep [options] pattern file
  • command | grep pattern

GREP - Essential Options

OptionDescription
-iCase insensitive
-vInvert (lines that do NOT match)
-nDisplay line number
-cCount matching lines
-oDisplay ONLY the matched part
-rRecursive in directories
-lDisplay only file names
-A n, -B n, -C nContext: After, Before, Combined

GREP -c: Caution!

$ echo "error error error" | grep -c 'error'
1 ‚Üê Counts LINES, not occurrences!
-c counts LINES containing the pattern, not the total number of occurrences!

To count ALL occurrences:

$ echo "error error error" | grep -o 'error' | wc -l
3 ‚Üê Correct!

SED - Stream Editor

Edits text in a stream - processes line by line without loading the entire file.

$ sed 's/old/new/' file.txt
... text with "new" instead of "old" ...
üí° Key insight: sed by default writes to stdout, it does NOT modify the original file!

SED - The Substitution Command

s/PATTERN/REPLACEMENT/FLAGS
FlagEffect
gGlobal - all occurrences on a line
iCase insensitive
nThe nth occurrence
$ echo "cat cat cat" | sed 's/cat/dog/'
dog cat cat ‚Üê Only the first!

$ echo "cat cat cat" | sed 's/cat/dog/g'
dog dog dog ‚Üê All of them!

SED - Edit In-Place

üö® DANGER: sed 's/x/y/' file > file
The shell empties the file BEFORE sed reads it!

Correct solutions:

# With backup (RECOMMENDED)
$ sed -i.bak 's/old/new/g' file.txt

# Without backup (DANGEROUS!)
$ sed -i 's/old/new/g' file.txt

AWK - Structured Processing

AWK treats each line as a record with separate fields.

$ echo "John Smith 30" | awk '{print $1}'
John

$ echo "John Smith 30" | awk '{print $3}'
30
Variables
$0 = the entire line
$1, $2... = fields
$NF = last field
NR = line number
Structure
BEGIN { } - before
{ actions } - per line
END { } - after

AWK - Print With and Without Comma

$ echo "a b" | awk '{print $1 $2}'
ab ‚Üê Direct concatenation!

$ echo "a b" | awk '{print $1, $2}'
a b ‚Üê With space (OFS)!
‚ö†Ô∏è Very common mistake!
The comma in print inserts the Output Field Separator (default: space).
Without comma = direct concatenation with no separator.

AWK - CSV Processing

# Set separator to comma
$ awk -F',' '{print $2}' employees.csv

# Skip header (NR > 1)
$ awk -F',' 'NR > 1 {print $2}' employees.csv

# Sum of salaries
$ awk -F',' 'NR > 1 {sum += $4} END {print sum}' employees.csv

Pipeline Magic

Combining grep, sed and awk creates powerful solutions:

# Top 5 IPs from log
$ awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -5

# Cleanup config
$ sed '/^#/d; /^$/d' config.txt | sort

# Unique errors with frequency
$ grep -i error log.txt | awk '{print $NF}' | sort | uniq -c | sort -rn

Summary

When to use what?
  • grep = search/filter
  • sed = replace/transform
  • awk = structured processing
Mistakes to avoid
  • grep without -E for +, ?
  • sed without /g
  • awk print without comma
  • -c counts lines, not matches
